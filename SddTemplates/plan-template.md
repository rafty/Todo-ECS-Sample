# plan-template.md

目的:
- `spec.md` をもとに、実装方針を整理した `plan.md` を作成する。

あなたが最初に行うこと:
- 対象 feature のディレクトリを確認する。
- 対象ディレクトリに最も近い `AGENTS.md` を確認する。
- 対象領域の `README.md` を確認する。
- `spec.md` を読み、要求と受け入れ条件を整理する。
- 必要に応じて `docs/` と既存コードを確認し、既存パターンとの整合を確認する。

入力:
- feature directory: `{{FEATURE_DIR}}`
- spec file: `{{FEATURE_DIR}}/spec.md`

出力:
- `{{FEATURE_DIR}}/plan.md`

作業ルール:
- `plan.md` では「どう作るか」を整理する。
- 要求の再定義ではなく、実装方針、分割方針、影響範囲、検証方針を書く。
- 推測は避け、未確認の点は「要確認」として扱う。
- 既存コードのパターンを優先し、新しい方式を導入する場合は理由を書く。
- 単一領域の feature か、複数領域にまたがる feature かを明記する。
- 実装は行わない。
- `tasks.md` はまだ作成しない。

`plan.md` に含める内容:
1. 実装方針
2. 変更対象
3. 変更しないもの
4. 技術方針
5. データや契約への影響
6. リスク
7. 検証方針
8. ドキュメント更新方針
9. 段階的な実施順序
10. 未解決事項

出力フォーマット:
以下の構成で `plan.md` を作成すること。

---

# Plan: {{FEATURE_NAME}}

## 実装方針
- この feature をどう実現するかを要約する。

## 変更対象
- 変更が入る想定のディレクトリ、モジュール、設定、契約を書く。

## 変更しないもの
- 今回は変更しない領域や対象外の変更を書く。

## 技術方針
- 既存パターンの再利用方針
- 新規クラスや新規コンポーネントが必要か
- 新規設定追加の要否
- 新規依存追加の要否
- 既存インフラや既存契約を参照するだけか、変更が必要か

## データや契約への影響
- DB スキーマ
- API 契約
- イベント契約
- 環境変数
- Secret / 設定値
- デプロイや運用への影響
  上記に該当があれば書く。

## リスク
- 破壊的変更の可能性
- 互換性への影響
- 運用影響
- セキュリティ影響
- 監視影響

## 検証方針
- 何をどのレベルで検証するかを書く。
- 例:
    - 単体テスト
    - 結合テスト
    - build
    - lint
    - typecheck
    - synth
    - validate
    - plan

## ドキュメント更新方針
- 更新が必要な `README.md`
- 更新が必要な `docs/`
- `docs/adr/` の要否

## 実施順序
1. 先に行うこと
2. 次に行うこと
3. 最後に行うこと

## 未解決事項
- まだ判断が必要なことを書く。

---

補足:
- 既存インフラを参照するだけなら、「参照のみで infra の変更は行わない」と明記する。
- 複数領域にまたがる場合は、領域ごとの責務分担を書く。
