# spec-template.md

目的:
- `requirements-draft.md` をもとに、実装前の feature 仕様書 `spec.md` を作成する。

あなたが最初に行うこと:
- 対象 feature のディレクトリを確認する。
- 対象ディレクトリに最も近い `AGENTS.md` を確認する。
- 対象領域の `README.md` を確認する。
- 必要に応じて `docs/` 配下の関連文書を確認する。
- 推測ではなく、既存コード・既存設定・既存文書に基づいて整理する。

入力:
- feature directory: `{{FEATURE_DIR}}`
- draft file: `{{FEATURE_DIR}}/requirements-draft.md`

出力:
- `{{FEATURE_DIR}}/spec.md`

作業ルール:
- `requirements-draft.md` の内容を整理し、曖昧な要求は「未確定事項」または「前提」に分ける。
- まだ決まっていないことを、決まっている事実のように書かない。
- 実装方法ではなく、何を満たすべきかを中心に書く。
- コード変更が単一領域か複数領域かを明記する。
- 既存の仕組みを利用する前提か、新規の構成変更が必要かを明記する。
- 必要に応じて、対象外を明記する。
- 実装は行わない。
- `plan.md` や `tasks.md` は作成しない。

`spec.md` に含める内容:
1. 概要
2. 背景
3. 目的
4. スコープ
5. 対象外
6. ユーザーストーリー / 利用シナリオ
7. 機能要件
8. 非機能要件
9. 受け入れ条件
10. 制約
11. 依存関係
12. 未確定事項 / 要確認事項

出力フォーマット:
以下の構成で `spec.md` を作成すること。

---

# Spec: {{FEATURE_NAME}}

## 概要
- この feature で実現することを簡潔に書く。

## 背景
- なぜこの feature が必要かを書く。

## 目的
- この feature の達成目標を書く。

## スコープ
- 今回対応する範囲を書く。

## 対象外
- 今回は対応しない内容を書く。

## ユーザーストーリー / 利用シナリオ
- 利用者や開発者の観点で、どのように使われるかを書く。

## 機能要件
- 要件を箇条書きで整理する。
- それぞれ曖昧さがないように書く。

## 非機能要件
- 性能、保守性、セキュリティ、運用、監視などの観点があれば書く。

## 受け入れ条件
- 完了とみなす条件を箇条書きで書く。

## 制約
- 技術的制約、運用上の制約、既存契約との整合などを書く。

## 依存関係
- 関連する領域、既存サービス、既存インフラ、既存 API などを書く。

## 未確定事項 / 要確認事項
- 現時点で確定していない点を書く。

---

補足:
- backend だけで閉じる feature なら、その前提を明記する。
- infra や frontend を参照するだけなら、その旨を書く。
- 他領域の変更が必要な場合は、その領域名を明記する。
