# tasks-template.md

目的:
- `spec.md` と `plan.md` をもとに、実装可能な粒度まで分解した `tasks.md` を作成する。

あなたが最初に行うこと:
- 対象 feature のディレクトリを確認する。
- 対象ディレクトリに最も近い `AGENTS.md` を確認する。
- 対象領域の `README.md` を確認する。
- `spec.md` と `plan.md` を読み、実装対象と非対象を整理する。
- 必要に応じて既存コードと `docs/` を確認し、タスクの粒度を現実的にする。

入力:
- feature directory: `{{FEATURE_DIR}}`
- spec file: `{{FEATURE_DIR}}/spec.md`
- plan file: `{{FEATURE_DIR}}/plan.md`

出力:
- `{{FEATURE_DIR}}/tasks.md`

作業ルール:
- `tasks.md` は実装順に並べる。
- 1 タスクは、できるだけ 1 つの目的に集中させる。
- タスクは実装可能で確認可能な粒度にする。
- 「調査」「実装」「テスト」「ドキュメント更新」を分ける。
- spec にない作業を勝手に追加しない。
- plan に反するタスクを入れない。
- 依存関係が分かるように並べる。
- 並列に進められるものは分かるようにする。
- 実装は行わない。

`tasks.md` に含める内容:
1. 前提確認タスク
2. 実装タスク
3. テスト / 検証タスク
4. ドキュメント更新タスク
5. 完了確認タスク

出力フォーマット:
以下の構成で `tasks.md` を作成すること。

---

# Tasks: {{FEATURE_NAME}}

## 前提確認
- [ ] 関連する `AGENTS.md`、`README.md`、`docs/` を確認する
- [ ] `spec.md` の未確定事項を確認する
- [ ] `plan.md` の変更対象と非対象を確認する

## 実装タスク
- [ ] タスク 1
- [ ] タスク 2
- [ ] タスク 3

## テスト / 検証タスク
- [ ] 関連する単体テストを追加または更新する
- [ ] 関連する build / lint / typecheck / test / synth / validate / plan を実行する
- [ ] 未実施の検証がある場合は理由を記録する

## ドキュメント更新タスク
- [ ] 必要な `README.md` を更新する
- [ ] 必要な `docs/` を更新する
- [ ] 必要に応じて `docs/adr/` を追加または更新する

## 完了確認
- [ ] 受け入れ条件を満たしていることを確認する
- [ ] 対象外の変更が混ざっていないことを確認する
- [ ] シークレットや不要ファイルが差分に含まれていないことを確認する

---

タスク分解ルール:
- タスク名は、何をするかが分かる動詞で始める。
- 1 つのタスクに複数の大きな目的を混ぜない。
- 実装タスクと検証タスクを分ける。
- backend / frontend / infra / o11y の複数領域にまたがる場合は、領域ごとに見出しを分ける。
- 既存インフラを参照するだけで変更しない場合は、infra の実装タスクは作らない。
