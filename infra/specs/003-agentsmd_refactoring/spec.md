# Spec: 003-agentsmd_refactoring

## 概要
- 既存の `infra/` 配下AWS CDK実装について、ルート `AGENTS.md` と `infra/AGENTS.md` のガイドラインに準拠するためのリファクタリング要件を定義する。
- ネットワーク要件自体の追加ではなく、既存実装の構成・責務分離・運用性・保守性をガイドライン整合の観点で是正する。

## 背景
- 先行要件でVPC等のネットワーク基盤をCDKで作成済みだが、ガイドライン認識不足により、`AGENTS.md` の方針（特に `infra/AGENTS.md` の設計・運用ルール）に沿わない実装が含まれている。
- ガイドライン参照経路は修正済みであり、今後の作業品質を安定化するため、既存コードを現行ガイドラインに合わせて再整理する必要がある。

## 目的
- `infra/` のCDKコードを、リポジトリ運用ルールとインフラ設計ルールに整合する状態へリファクタリングする。
- 以降の機能追加（ECS/ALB/DB等）で再利用しやすい構造とし、責務分離・最小変更・安全性を満たす。

## スコープ
- 対象領域: `infra/`（AWS CDK TypeScript）
- 本仕様で定義する対象:
  - 既存ネットワーク関連CDK実装のガイドライン準拠化
  - Stack/Construct責務の見直し（必要最小限の分割・整理）
  - 環境切替・タグ付与・コマンド運用など既存ルールとの整合確認
  - 変更範囲に応じた `synth` を中心とする検証要件の明確化

## 対象外
- 新規AWSサービス（ECSサービス、ALB、Aurora詳細設定、CloudFront等）の追加実装
- 本質的に要件外となる大規模な横断リファクタリング、広範囲rename、ディレクトリ移動
- 本番影響を伴うデフォルト値の変更（明示依頼がないもの）

## ユーザーストーリー / 利用シナリオ
- インフラ担当者として、`infra/` の既存CDKコードが `AGENTS.md` に準拠していることを前提に、安全に機能追加したい。
- レビュアーとして、Stack責務・Construct分割・命名・検証手順が明確で、差分の妥当性を短時間で判断したい。
- 運用担当者として、環境指定やタグ方針が統一され、`synth`/`diff` で変更影響を追跡しやすい状態を維持したい。

## 機能要件
- ガイドライン準拠
  - `infra/` の修正はルート `AGENTS.md` と `infra/AGENTS.md` の両方に準拠すること。
  - 変更は依頼範囲に限定し、不要な無関係差分を含めないこと。
- 設計・実装整合
  - Stackは単一責務を維持し、必要に応じて `lib/constructs/` へ再利用単位を切り出すこと。
  - Constructはインフラ構成要素の表現に限定し、業務ロジックを含めないこと。
  - 既存の構成・命名・環境設定パターン（`context` / `config` / `props`）に合わせること。
  - AWS CDKはL2/L3 Construct優先とし、L1使用は要件上必要な場合に限定すること。
- 既存ネットワーク実装の是正
  - 既存ネットワーク関連コードに対し、責務過多・再利用性不足・可読性低下など、ガイドライン不整合箇所を是正すること。
  - 既存要件で定義済みの環境切替（`-c env=<dev|stg|prod>`）やタグ方針を破壊しないこと。
- 検証
  - 変更後に `infra/` の既存フローに従って、少なくとも `synth` 相当の検証を実施できる状態であること。
  - 影響確認が重要な変更では `diff` による差分確認を行えること。

## 非機能要件
- 保守性
  - 将来のインフラ拡張時に、Stack/Constructの責務境界が明確であること。
  - 変更箇所が追跡しやすく、レビューしやすい差分粒度を維持すること。
- 安全性
  - 高リスク変更（IAM、ネットワーク、Security Group、ロール/ポリシー）は最小化し、必要時は変更理由を明確にすること。
  - 破壊的なrename/削除/置換を安易に行わないこと。
- 運用性
  - 既存コマンド（`build` / `test` / `synth` / `diff`）との整合を維持すること。
  - 環境識別・サービス識別・バージョントレースのタグ運用を継続可能であること。

## 受け入れ条件
- `infra/` の対象コード差分が、ガイドライン準拠化に必要な範囲へ限定されている。
- Stack/Constructの責務境界が明確化され、既存の環境切替・命名・構成方針と整合している。
- 変更した領域に対するインフラ検証（最低 `synth`）結果を提示できる、または未実施理由が明確である。
- ドキュメント更新要否（`infra/README.md`、`docs/infra/`、必要時 `docs/adr/`）が確認されている。

## 制約
- 実装対象は `infra/` に限定する。
- ガイドライン修正内容を前提とし、要件定義時点で未確定の事項は確定情報として扱わない。
- 既存のプロジェクト構成・命名・運用フローを尊重し、推測による仕様追加を行わない。

## 依存関係
- ルート `AGENTS.md`（全体ルール）
- `infra/AGENTS.md`（インフラ領域ルール）
- `infra/README.md`（主要コマンド、環境切替ルール、既存ネットワーク前提）
- 既存ネットワーク仕様・実装（`infra/specs/002-create_network/` と `infra/lib/`）

## 未確定事項 / 要確認事項
- 「ガイドラインに沿っていないコード」の具体的対象範囲（ファイル・観点）の最終確定。
- `synth` 以外に必須とする検証深度（`test` や `diff` の必須条件）の合意。
- 今回のリファクタリングで許容する構造変更の上限（例: Construct新設の可否、既存ファイル分割の許容範囲）。
