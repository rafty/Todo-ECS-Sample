# Plan: 003-agentsmd_refactoring

## 実装方針
- 単一領域（`infra/`）で、既存ネットワーク関連CDK実装を `AGENTS.md` / `infra/AGENTS.md` の運用・設計ルールに整合させる。
- 要件追加ではなく是正リファクタリングを目的とし、変更は最小限・非破壊で進める。
- 既存の環境切替（`-c env=<dev|stg|prod>`）・タグ運用（`env`/`service`/`version`）・コマンドフロー（`build`/`test`/`synth`/`diff`）を維持した上で、責務境界と可読性を改善する。

## 変更対象
- `infra/lib/` の既存ネットワーク関連 Stack 実装（責務過多・可読性・分割粒度の是正対象）。
- 必要に応じた `infra/lib/constructs/` への Construct 抽出（再利用性と単一責務を満たす範囲で実施）。
- `infra/bin/` や `infra/lib/config/` の環境切替・設定参照経路（破壊しないことを前提に整合確認）。
- 変更内容に応じた `infra/README.md`、`docs/infra/`、必要時 `docs/adr/` の更新有無確認。

## 変更しないもの
- 新規AWSサービス（ECS/ALB/CloudFront/Aurora詳細設定等）の追加実装。
- `infra/` 以外（`backend/`、`frontend/`、ルート全体）への機能変更。
- 明示依頼のない本番影響設定の変更、広範囲rename、ディレクトリ移動、大規模横断リファクタリング。

## 技術方針
- 既存パターンの再利用方針
  - 既存の Stack 構成、命名規則、`context` / `config` / `props` の扱いを踏襲する。
  - 既存の共通化済み Construct / helper がある場合は再利用を優先する。
- 新規クラスや新規コンポーネント
  - Stack の責務分離に必要な場合のみ、`infra/lib/constructs/` に意味のある単位で Construct を追加する。
  - Construct は IaC 構成要素に限定し、業務ロジックを持たせない。
- 新規設定追加
  - 原則追加しない。既存設定で要件を満たせない場合のみ最小限追加し、理由を明確化する。
- 新規依存追加
  - 原則追加しない。既存CDK/L2-L3 Constructで実現し、L1はやむを得ない場合のみ最小範囲で使用する。
- 既存インフラや既存契約との関係
  - 既存ネットワーク仕様（`infra/specs/002-create_network/`）を参照し、挙動互換を維持する。

## データや契約への影響
- DBスキーマ: 変更なし。
- API契約: 変更なし。
- イベント契約: 変更なし。
- 環境変数: 追加なしを原則とし、既存の環境指定方式を維持する。
- Secret / 設定値: 変更なし。
- デプロイや運用への影響:
  - `synth` / `diff` でテンプレート差分を確認し、論理構成の変更有無を可視化する。
  - リソース置換・削除を伴う差分が出た場合は高リスクとして再評価する。

## リスク
- 破壊的変更の可能性
  - Stack/Construct 分割時の論理ID変化による置換リスク。
- 互換性への影響
  - 既存の環境切替・タグ付与が崩れると運用互換性が失われる。
- 運用影響
  - 構造変更によりレビュー/保守の参照箇所が変化する。
- セキュリティ影響
  - IAM/ネットワーク/SG 関連に意図しない差分が混入するリスク。
- 監視影響
  - 監視対象リソースの論理ID・構成変化がある場合、既存運用手順との整合確認が必要。

## 検証方針
- `infra/` の既存スクリプトに従い、以下を段階的に実施する。
  - `npm run build`（TypeScriptコンパイル整合確認）
  - `npm test -- --runInBand`（既存テストがある場合の回帰確認）
  - `npx cdk synth -c env=prod`（必須: テンプレート生成可否）
  - `npx cdk diff -c env=prod`（影響確認が重要な差分時に実施）
- 検証不能項目がある場合は、未実施理由と未確認範囲を明示する。

## ドキュメント更新方針
- `infra/README.md`
  - 主要コマンド、構成説明、タグ運用、環境切替ルールに変更が生じる場合のみ更新する。
- `docs/infra/`
  - ネットワーク構成や運用フローの説明に変更が生じる場合のみ更新する。
- `docs/adr/`
  - 設計判断（責務分離方針、運用ルール変更）が新規・更新を要する場合に追記を検討する。

## 実施順序
1. 現状把握
   - `infra/lib/` の既存実装を点検し、ガイドライン不整合箇所を抽出する。
   - 影響範囲を `infra/` 内で確定し、対象外との差分境界を明確化する。
2. 設計確定
   - Stack/Construct の責務境界を定義し、分割が必要な最小単位を決定する。
   - 既存の環境切替・タグ運用を維持するための変更方針を確定する。
3. 実装修正
   - 必要最小限のコード修正を行い、命名・配置・再利用方針を既存規約に合わせる。
   - 高リスク領域（IAM/ネットワーク/SG）は変更理由を明確化して慎重に反映する。
4. 検証
   - `build` / `test` / `synth` を実行し、必要時 `diff` で影響確認する。
   - 差分内容をレビューし、不要な変更が混入していないことを確認する。
5. ドキュメント・仕上げ
   - README/docs/ADR の更新要否を判定し、必要時のみ日本語で更新する。
   - 完了条件（範囲限定・検証結果・差分妥当性）を満たすことを最終確認する。

## 未解決事項

- `diff` を必須化する条件（例: 論理ID変化、ネットワーク/IAM/SG変更を含む場合）の最終合意。
- 今回許容する構造変更の上限（Construct新設数、既存ファイル分割の許容範囲）。
- `version` タグ運用の厳密ルール（現時点でハッシュ算出を採用するか、固定値運用とするか）の運用確定。
