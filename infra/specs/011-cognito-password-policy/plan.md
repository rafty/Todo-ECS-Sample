# Plan: 011-cognito-password-policy

## 実装方針
- 本 feature は **複数領域（`infra/` + `docs/`）** で実施する。
- Cognito の既存方針（`prod` で自己登録可、管理者作成可）は維持し、`tempPasswordValidity` 無期限化は行わない。
- 仮パスワード失効リスクは、負荷試験用ユーザー作成時に `AdminSetUserPassword (Permanent=true)` を組み合わせる運用で回避する。
- 主要成果物は `docs/` 配下の運用手順書とし、必要最小限の IaC 差分で整合を取る。

## 技術方針
- 既存パターンの再利用方針
  - Cognito は既存の `TodoCognitoConstruct`（L2 Construct）を継続利用する。
  - 環境切替は既存の `-c env=<dev|stg|prod>` を踏襲し、対象運用は `prod` 固定で記述する。
- 大量ユーザー作成の実行方針
  - `AdminCreateUser` に `MessageAction=SUPPRESS` を適用し、招待メール送信を抑止する。
  - 作成直後に `AdminSetUserPassword --permanent` を実行し、仮パスワード期限の影響を排除する。
  - 削除は `AdminDeleteUser` で実行する。
  - 一覧確認は `ListUsers`（必要に応じてページング）を用いる。
- ユーザー識別方針
  - 件数上限は設けないが、標準運用は約100ユーザーとする。
  - `0001` 形式の番号管理IDは運用上の識別子として使用し、Cognito 登録時は email 形式制約に合わせてマッピングする（例: `loadtest-0001@test.local`）。
  - 負荷試験用ユーザーの email ドメインは実在ドメインではなく、テスト用ドメインを使用する。
- パスワード方針
  - 大量ユーザー作成時に設定するパスワードは固定値（全ユーザー共通）とする。
  - 固定パスワード運用のため、追加の保管方式は設けない前提とする。
- IAM 方針
  - バルク作成・削除は Admin 権限相当の実行ロールで行う。
  - 実運用文書には対象 User Pool に限定した IAM 権限例を記載する。
- 新規依存追加
  - 依存パッケージの追加は行わない。CLI コマンドと既存 CDK だけで運用を成立させる。

## データや契約への影響
- DB スキーマ: 影響なし。
- API 契約: 影響なし。
- イベント契約: 影響なし。
- 環境変数: 追加なし。
- Secret / 設定値:
  - アプリ用 Secret への変更なし。
  - 負荷試験用パスワードは固定値で運用し、追加の保管方式は設けない。
- デプロイや運用への影響:
  - IaC 変更がない場合、デプロイ影響は軽微。
  - 運用手順導入により、負荷試験前準備（作成/再作成/削除）が標準化される。

## リスク
- 運用リスク
  - Cognito API のスロットリング（`TooManyRequestsException`）でバッチ処理が失敗する可能性。
  - 対策: 1秒あたり約100件を上限として実行し、超過時は低レート化とリトライ方針を手順に明記する。
- セキュリティリスク
  - Admin 権限相当の誤用で不要ユーザー作成・削除が発生する可能性。
  - 対策: 対象 User Pool 制限、実行ロール限定、実行ログ保全を手順化する。
- 互換性リスク
  - ユーザー識別子 `0001` の扱いが Cognito の sign-in 属性制約と不整合になる可能性。
  - 対策: 運用識別子と Cognito 登録値の対応規則を明文化する。
- 破壊的変更リスク
  - User Pool 設定（自己登録可否、パスワードポリシー）を不用意に変更すると既存ログイン導線へ影響。
  - 対策: IaC 差分は最小化し、必要時のみ `synth/diff` で確認する。

## 検証方針
- ドキュメント検証
  - 手順書に記載したコマンドが `prod` 前提と整合していることをレビューで確認する。
  - 作成・再実行・削除・例外時対応の一連手順が欠けなく記載されていることを確認する。
  - テスト用ドメイン利用、固定パスワード運用、1秒あたり約100件上限が手順に反映されていることを確認する。
- `infra/` 検証（IaC差分がある場合）
  - `npm run build`
  - `npm test -- --runInBand`
  - `npx cdk synth -c env=prod`
  - 影響がある場合のみ `npx cdk diff -c env=prod`
- 実行不能時の扱い
  - AWS 認証や権限不足で検証できない場合は、未確認範囲と理由を明記する。
