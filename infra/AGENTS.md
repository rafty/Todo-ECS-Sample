# AGENTS.md

## 対象範囲
- このファイルは `infra/` 配下に適用されます。

## 最初に確認すること
- `infra/README.md` を確認し、この領域の概要、主要コマンド、関連ドキュメントを把握してください。
- 詳細なインフラ構成、環境差分、デプロイフローが必要な場合は `docs/infra/` を確認してください。

## 作業ルール
- インフラ変更は最小限かつ意図的に行ってください。
- Stack は**単一責任（SRP）**を徹底してください。1つの Stack は、Network、App、DB、Monitoring など、明確な 1 つの目的に絞ってください。
- 既存のスタック構成、Construct の分割、命名規則、環境設定の持たせ方を維持してください。
- `lib/` 配下の Stack / Construct でコードが肥大化しないようにしてください。
- 再利用可能な単位は Construct として切り出し、`lib/constructs/` 配下に配置してください。
- Construct の命名は、役割が分かる明確な名前にしてください。
- 共通化済みの Construct や helper がある場合は再利用を優先してください。
- 環境ごとの差異は、既存の `context`、`config`、`props` の持たせ方に合わせてください。
- リソースの rename、置換、削除は高リスクとして扱ってください。
- IAM、ネットワーク、セキュリティグループ、ロール、ポリシーの変更は高リスクとして慎重に扱ってください。
- 権限は必要最小限にしてください。
- タスクで必要ない限り、アプリ変更とインフラ変更を混在させないでください。

## Construct 設計
- 再利用されるパターンは `infra/lib/constructs/` に配置してください。
- Stack ファイル (`infra/lib/*-stack.ts`) にリソース定義や組み立て処理を過剰に詰め込まず、適切な粒度で Construct に分割してください。
- 複数のリソースを意味のある単位でまとめて再利用できる場合は、Construct として切り出してください。
- Construct は「何をまとめた単位なのか」が分かる責務で設計してください。
- Construct 内に**業務ロジックを含めないでください**。Construct は IaC の構成要素の表現に限定してください。
- 既存の Stack が肥大化している場合は、変更に必要な範囲でのみ安全に Construct へ分割してください。
- 単純な共通処理や設定値の共通化と、意味のあるインフラ構成単位としての Construct は区別して設計してください。

## L2 / L3 Construct 優先
- AWS CDK では **L2 / L3 Construct を優先**して使用してください。
- **L1 Construct は原則使用しないでください。**
- L1 Construct を使う必要がある場合は、既存の L2 / L3 で要件を満たせない理由をコードコメントで明記してください。
- L1 Construct を使う場合でも、利用範囲は必要最小限にしてください。

## コメント方針
- コードの処理単位ごとに、**「なぜ必要か」**を日本語コメントで記載してください。
- コードに注意点がある場合は、必ず日本語でコメントを書いてください。
- 曖昧な要件や危険な要件がある場合は、保守的な実装を優先し、必要に応じて以下の形式でコメントを追加してください。
    - `// TODO: 人間に確認が必要: <内容>`

## コマンド
- ロックファイルに対応するパッケージマネージャを使用してください。
- `install`、`build`、`test`、`synth`、`diff` は既存の script を優先してください。
- 専用 script がない場合は、このリポジトリで採用されている CDK の既存フローに従ってください。
- インフラ変更では `synth` を実行し、影響確認が重要な場合は `diff` も実行してください。

## README / docs 更新ルール
- 起動方法、主要コマンド、重要なディレクトリ構成の説明が変わる場合は `infra/README.md` を更新してください。
- インフラ構成、環境差分、デプロイフローが変わる場合は `docs/infra/` を更新してください。
- システム全体の構成や責務分担が変わる場合は `docs/architecture/` を更新してください。
- 共通の開発手順や前提ツールが変わる場合は `docs/development/` を更新してください。
- 重要なインフラ判断の追加・変更がある場合は `docs/adr/` を更新または追加してください。
- ドキュメントは必ず日本語で生成・更新してください。
- 構成図があると分かりやすい場合は、文章だけでなく構成図（Mermaid）も記載してください。

## スタイル & 品質ツール
- 以下のツールがプロジェクトに設定されている場合、必ず尊重してください。
- ESLint: ルールを無効化せず、修正で対応してください。
- Prettier: フォーマットを遵守してください。
- cdk-nag: 警告が出た場合、無視せず修正するか、理由を明記してください。

## 検証
- 変更したファイルに関連するインフラ検証を実行してください。
- `synth` を含め、影響確認が重要な場合は `diff` も確認してください。
- 検証を実行できない場合は、未確認の内容と理由を明記してください。

## 完了条件
- 差分が意図したインフラ変更に限定されていること。
- 関連するインフラ検証が完了していること、または未確認範囲が明確であること。
- 必要な `infra/README.md`、`docs/infra/`、関連 docs の更新要否を確認していること。

## 禁止事項
- シークレット、認証情報、秘密鍵をコードに埋め込まないでください。
- 明示的な依頼なしに本番影響のある既定値を変更しないでください。
- 破壊的な rename や削除を安易に行わないでください。
