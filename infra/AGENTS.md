# AGENTS.md

## 最初に確認すること
- `infra/README.md` を確認し、この領域の概要、主要コマンド、関連ドキュメントを把握してください。
- 詳細なインフラ構成、環境差分、デプロイフローが必要な場合は `docs/infra/` を確認してください。

## 作業ルール
- インフラ変更は最小限かつ意図的に行ってください。
- Stack は **単一責任（SRP）** を徹底すること。1つの目的とは、Network, App, DB, Monitoringなどです。
- 既存のスタック構成、construct の分割、命名規則、環境設定の持たせ方を維持してください。
- `lib/`のStack、Constructで、コードが肥大化しないように再利用可能なConstructを作成し`lib/constructs/`のコンストラクトを使用すること。また、コンストラクトのネーミングはわかりやすいものにすること。
- 共通化済みの construct や helper がある場合は再利用を優先してください。
- 環境ごとの差異は、既存の context、config、props の持たせ方に合わせてください。
- リソースの rename、置換、削除は高リスクとして扱ってください。
- IAM、ネットワーク、セキュリティグループ、ロール、ポリシーの変更は高リスクとして慎重に扱ってください。
- 権限は必要最小限にしてください。
- タスクで必要ない限り、アプリ変更とインフラ変更を混在させないでください。
- コードの処理単位毎に、必ずコメントに「なぜ必要か」を日本語で書いてください。
- コードに注意点があれば、必ず日本語でコメントを書いてください。
- コードに曖昧・危険な要件がある場合、`// TODO: 人間に確認が必要: <内容>` を追加し保守的な実装を優先してください。

## コマンド
- ロックファイルに対応するパッケージマネージャを使用してください。
- install、build、test、synth、diff は既存の script を優先してください。
- 専用 script がない場合は、このリポジトリで採用されている CDK の既存フローに従ってください。
- インフラ変更では synth を実行し、影響確認が重要な場合は diff も実行してください。

## README / docs 更新ルール
- 起動方法、主要コマンド、重要なディレクトリ構成の説明が変わる場合は `infra/README.md` を更新してください。
- インフラ構成、環境差分、デプロイフローが変わる場合は `docs/infra/` を更新してください。
- システム全体の構成や責務分担が変わる場合は `docs/architecture/` を更新してください。
- 共通の開発手順や前提ツールが変わる場合は `docs/development/` を更新してください。
- 重要なインフラ判断の追加・変更がある場合は `docs/adr/` を更新または追加してください。
- ドキュメントは必ず日本語で生成・更新してください。
- 構成図があるとわかりやすい場合は、文章よりも構成図（Mermaid）を記載して文章を補足してください。

## スタイル & 品質ツール
以下のツールがプロジェクトに設定されている場合、 必ず尊重してください。
- ESLint：ルールを無効化せず、修正すること
- Prettier：フォーマットを遵守
- cdk-nag：警告が出た場合、無視せず修正または理由を記載

## 検証
- 変更したファイルに関連するインフラ検証を実行してください。
- synth を含め、影響確認が重要な場合は diff も確認してください。
- 検証を実行できない場合は、未確認の内容と理由を明記してください。

## 完了条件
- 差分が意図したインフラ変更に限定されていること。
- 関連するインフラ検証が完了していること、または未確認範囲が明確であること。
- 必要な `infra/README.md`、`docs/infra/`、関連 docs の更新要否を確認していること。

## 禁止事項
- シークレット、認証情報、秘密鍵をコードに埋め込まないでください。
- 明示的な依頼なしに本番影響のある既定値を変更しないでください。
- 破壊的な rename や削除を安易に行わないでください。
