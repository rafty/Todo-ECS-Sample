# ADR 001: プロジェクト構成

## ステータス
採用

## コンテキスト
このプロジェクトは Todo アプリケーションであり、以下 3 つの構成要素を持つ。

- Spring Boot による backend アプリケーション
- React による frontend アプリケーション
- AWS リソースを作成し、backend と frontend をデプロイする AWS CDK（TypeScript）

これらは役割と技術スタックが異なるが、1 つのサンプルシステムとして一体管理したい。  
そのため、リポジトリの分割粒度とディレクトリ構成を決定する必要があった。

## 決定事項
Git 管理のルートを `Todo-ECS-Sample` とし、以下 3 つのサブプロジェクトを配置するモノレポ構成を採用する。

1. `backend/`  
   Spring Boot アプリケーションを配置する。

2. `frontend/`  
   React アプリケーションを配置する。

3. `infra/`  
   AWS CDK（TypeScript）プロジェクトを配置し、AWS リソース作成と backend / frontend デプロイを担う。

ディレクトリ構成:

```text
Todo-ECS-Sample/   <- ルートプロジェクト (git管理)
 │
 ├── backend/      <- Spring Boot プロジェクト
 │
 ├── frontend/     <- React プロジェクト
 │
 ├── infra/        <- AWS CDK (TypeScript) プロジェクト
 │
 ├── README.md
 └── .gitignore
```

## 根拠

- **単一リポジトリで全体を把握しやすくするため**  
  backend / frontend / infra を 1 つのリポジトリで管理すると、構成全体を一元的に追跡できる。
- **役割ごとに責務を分離するため**  
  アプリ実装とインフラ定義は変更対象が異なるため、ディレクトリ単位で分離した方が見通しがよい。
- **変更管理をしやすくするため**  
  1 つの機能で複数レイヤーに変更が入る場合でも、同一リポジトリ内で関連差分を管理しやすい。
- **サンプルとして扱いやすくするため**  
  利用者は 1 つのリポジトリ取得でアプリと AWS 構成をまとめて参照できる。
- **技術スタックごとの独立性を保つため**  
  backend / frontend / infra は独立したビルド・実行単位であり、サブプロジェクト分割と相性がよい。

## 影響

- プロジェクト全体を 1 つの Git リポジトリで管理できるため、変更追跡が容易になる。
- backend / frontend / infra の責務が明確になり、ディレクトリ単位で作業しやすくなる。
- 異なる技術スタックを併存させるため、ルート README と開発ルール整備が重要になる。
- サブプロジェクトごとに依存・実行方法が異なるため、各 README の継続的な保守が必要になる。
- 将来の CI/CD 設計もこの構成を前提に検討することになる。
