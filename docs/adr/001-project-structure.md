# ADR 001: プロジェクト構成

## ステータス
採用

## コンテキスト
このプロジェクトはTodoアプリケーションです。
対象となる構成要素は以下の 3 つである。

- Spring Boot による backend アプリケーション
- React による frontend アプリケーション
- AWS リソースを作成し、backend と frontend をデプロイする AWS CDK（TypeScript）

これらは役割や利用技術が異なる一方で、1 つのサンプルシステムとして一体で管理したい。  
そのため、リポジトリをどの粒度で分割するか、またどのようなディレクトリ構成で管理するかを決定する必要があった。

## 決定事項
全体の Git 管理は `Todo-ECS-Sample` をルートプロジェクトとして行い、その配下に以下の 3     つのサブプロジェクトを配置するモノレポ構成とします。

1. `backend/`  
   Spring Boot アプリケーションを配置する

2. `frontend/`  
   React アプリケーションを配置する

3. `infra/`  
   AWS CDK（TypeScript）プロジェクトを配置し、AWS リソースの作成と backend / frontend のデプロイを担う

ディレクトリ構成は以下とする。

```text
Todo-ECS-Sample/   <- ルートプロジェクト (git管理)
 │
 ├── backend/      <- Spring Boot プロジェクト
 │
 ├── frontend/     <- React プロジェクト
 │
 ├── infra/        <- AWS CDK (TypeScript) プロジェクト
 │
 ├── README.md
 └── .gitignore
````

## 根拠

* **単一リポジトリで全体を把握しやすくするため**: backend、frontend、infra を 1 つの Git リポジトリで管理することで、サンプル全体の構成を一元的に把握しやすくなるため。
* **役割ごとに責務を分離するため**: アプリケーション実装、AWS インフラ構築 では使用技術や変更対象が異なるため、ディレクトリ単位で責務を分けた方が見通しがよいため。
* **変更管理をしやすくするため**: 1 つの機能追加や検証で複数レイヤーに変更が入る場合でも、同一リポジトリ内で関連差分をまとめて管理しやすいため。
* **サンプルプロジェクトとして扱いやすくするため**: 利用者が 1 つのリポジトリを取得するだけで、アプリケーション、AWS 環境 をまとめて参照できるため。
* **技術スタックごとの独立性を保つため**: backend、frontend、infra はそれぞれ独立したビルド・実行単位として扱えるため、サブプロジェクトとして分ける構成が適しているため。

## 影響

* プロジェクト全体を 1 つの Git リポジトリで管理できるため、構成全体の把握と変更追跡がしやすくなる。
* backend、frontend、infra、の責務が明確になり、ディレクトリ単位で作業しやすくなる。
* 異なる技術スタックを 1 つのリポジトリで扱うため、ルートレベルの README や開発ルールの整理が重要になる。
* サブプロジェクトごとに依存関係や実行方法が異なるため、それぞれのセットアップ手順を明確に文書化する必要がある。
* 将来的に CI/CD や開発ルールを導入する場合も、このディレクトリ構成を前提に設計することになる。

## 備考

将来的に、以下の拡張を検討できる。

* 各サブプロジェクトごとの README 追加
* ルートプロジェクトでの共通開発ルールやタスク実行方法の整備
* Monorepo 向けツールの導入検討
* backend / frontend / infra ごとの CI 設計
