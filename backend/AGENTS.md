# AGENTS.md

## 対象範囲
- このファイルは `backend/` 配下に適用されます。

## 最初に確認すること
- `backend/README.md` を確認し、この領域の概要、主要コマンド、関連ドキュメントを把握してください。
- 詳細な API、外部連携、設計説明が必要な場合は `docs/backend/` を確認してください。

## 作業ルール
- 既存のパッケージ構成、レイヤ構成、命名規則、実装スタイルに従ってください。
- Controller、Service、Repository などの責務分離を維持してください。
- 依存注入は既存方針に従ってください。周辺コードに明確な別方針がない限り、一般にコンストラクタインジェクションを優先してください。
- バリデーション、例外処理、ログ出力は既存パターンに合わせてください。
- API 契約の変更は必要最小限に留め、影響範囲を明確にしてください。
- DB スキーマやマイグレーションの変更は、互換性と既存データへの影響を意識してください。
- 認証、認可、セキュリティ関連の変更は高リスクとして慎重に扱ってください。
- 無関係なクラスや設定への波及を避けてください。
- コードの処理単位毎に、必ずコメントに「なぜ必要か」を日本語で書いてください。
- コードに注意点があれば、必ず日本語でコメントを書いてください。
- コードに曖昧・危険な要件がある場合、`// TODO: 人間に確認が必要: <内容>` を追加し保守的な実装を優先してください。

## コマンド
- `gradlew` がある場合は `./gradlew` を優先してください。
- `mvnw` がある場合は `./mvnw` を優先してください。
- ad-hoc なコマンドより、既存のスクリプトや wrapper を優先してください。
- まず変更箇所に近い最小限のテストを実行し、必要に応じて範囲を広げてください。

## README / docs 更新ルール
- 起動方法、主要コマンド、重要なディレクトリ構成の説明が変わる場合は `backend/README.md` を更新してください。
- API、外部連携、永続化仕様、バックエンド設計が変わる場合は `docs/backend/` を更新してください。
- 開発フローや環境構築に共通影響がある場合は `docs/development/` を更新してください。
- 重要な技術判断の追加・変更がある場合は `docs/adr/` を更新または追加してください。
- ドキュメントは 必ず日本語で生成・更新してください。
- 構成図があるとわかりやすい場合は、文章よりも構成図（Mermaid）を記載して文章を補足してください。

## 検証
- 変更したファイルに関連するバックエンド検証を実行してください。
- API、外部連携、永続化に関わる変更では、それに対応する検証を含めてください。
- 検証を実行できない場合は、未確認の内容と理由を明記してください。

## 完了条件
- 変更が既存のバックエンド構造と整合していること。
- 関連するバックエンド検証が完了していること、または未確認範囲が明確であること。
- 必要な `backend/README.md`、`docs/backend/`、関連 docs の更新要否を確認していること。

## 禁止事項
- シークレット、認証情報、接続情報をコードに埋め込まないでください。
- 公開 API の挙動を意図せず変更しないでください。
- 本番向け設定を推測で変更しないでください。
