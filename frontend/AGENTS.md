# AGENTS.md

## 対象範囲
- このファイルは `frontend/` 配下に適用されます。

## 最初に確認すること
- `frontend/README.md` を確認し、この領域の概要、主要コマンド、関連ドキュメントを把握してください。
- 詳細な画面仕様、UI 方針、画面遷移が必要な場合は `docs/frontend/` を確認してください。

## 作業ルール
- 既存のコンポーネント構成、命名規則、スタイル方針、状態管理方針に従ってください。
- UI 変更は依頼された要件に必要な範囲に限定してください。
- 新しいコンポーネントや hook を作る前に、既存の共通部品やユーティリティの再利用を検討してください。
- props、state、hooks の責務を明確に保ってください。
- 型安全性を維持し、強い理由がない限り `any` を導入しないでください。
- API 連携の変更は、バックエンド契約との整合を保ってください。
- フォーム、バリデーション、ローディング、エラー表示は既存パターンに合わせてください。
- アクセシビリティを悪化させる変更は避けてください。
- 明示的な依頼がない限り、無関係な UI リニューアルや大規模リファクタは避けてください。
- 依存パッケージの追加は、必要性が明確な場合に限ってください。
- コードの処理単位毎に、必ずコメントに「なぜ必要か」を日本語で書いてください。
- コードに注意点があれば、必ず日本語でコメントを書いてください。
- コードに曖昧・危険な要件がある場合、`// TODO: 人間に確認が必要: <内容>` を追加し保守的な実装を優先してください。

## コマンド
- ロックファイルに合わせてパッケージマネージャを選んでください。
    - `pnpm-lock.yaml` がある場合は `pnpm`
    - `yarn.lock` がある場合は `yarn`
    - `package-lock.json` がある場合は `npm`
- lint、test、typecheck、build は既存の script を優先してください。
- まず変更箇所に近い最小限の検証を実行し、必要に応じて範囲を広げてください。


## README / docs 更新ルール
- 起動方法、主要コマンド、重要なディレクトリ構成の説明が変わる場合は `frontend/README.md` を更新してください。
- 画面挙動、画面遷移、UI ガイド、ユーザー向け挙動が変わる場合は `docs/frontend/` を更新してください。
- 開発フローや環境構築に共通影響がある場合は `docs/development/` を更新してください。
- バックエンド契約に依存する変更がある場合は `docs/backend/` との整合も確認してください。
- 重要な設計判断の追加・変更がある場合は `docs/adr/` を更新または追加してください。
- ドキュメントは 必ず日本語で生成・更新してください。
- 構成図があるとわかりやすい場合は、文章よりも構成図（Mermaid）を記載して文章を補足してください。

## 検証
- 変更したファイルに関連するフロントエンド検証を実行してください。
- 型、ルーティング、共有部品、バンドルに影響する変更では、typecheck や build を含めてください。
- 検証を実行できない場合は、未確認の内容と理由を明記してください。

## 完了条件
- 変更が依頼された UI / フロントエンド範囲に収まっていること。
- 関連するフロントエンド検証が完了していること、または未確認範囲が明確であること。
- 必要な `frontend/README.md`、`docs/frontend/`、関連 docs の更新要否を確認していること。

## 禁止事項
- シークレット、`.env` の実値、個人用トークンをコミットしないでください。
- 明示的な依頼なしに大規模な UI 全面改修を行わないでください。
- 必要性が不明な依存追加を行わないでください。
